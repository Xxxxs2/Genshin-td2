# 千星奇域船塔防搭建蓝图

这份文档用于把当前 Godot 原型翻译成千星奇域 UGC 编辑器里可以实际搭建的规则系统。当前方向是“纵向长海图 + 船头自动航行 + 玩家控制航向和加速 + 多路线弯折海道 + 岛屿地形限制 + 岛上炮台锁定攻击 + 船自动攻击 + 过关后三选一强化”。

本文把内容分为三层：

- **确定玩法层**：无论编辑器具体组件名称如何变化，都应保留的游戏规则。
- **UGC 实现层**：使用变量、触发区域、计时器、对象启停和伤害组件搭建的推荐方案。
- **降级方案层**：当编辑器缺少载具控制、自定义投射物或自定义 UI 时的替代实现。

千星奇域仍可能更新编辑器能力和组件名称，因此本文不会假设某个未验证接口一定存在。开始正式制作前，先在当前版本编辑器内完成“能力验证关卡”，再决定最终实现方式。

## 文档导航

1. **首日能力验证**：先确认编辑器能不能做。
2. **UGC 总体架构**：确定船、航段和对象如何组织。
3. **变量设计**：建立全局、航船、武器和炮台变量。
4. **具体触发器实现**：逐个搭建移动、攻击、路线、强化和失败流程。
5. **推荐制作步骤**：按阶段从测试房做到可发布版本。
6. **性能与玩法规格**：控制对象预算，并查看视觉、数值和地图方案。
7. **后续扩展方案**：增加环境、强化质变、首领、模块随机和多人。
8. **发布检查**：确认稳定性、视觉版权和首版范围。

## 一、首日能力验证

不要一开始就搭完整长地图。先制作一个空白测试房，只验证以下九项：

### 1.1 编辑器能力检查表

| 检查项 | 最小验证方式 | 通过标准 | 不支持时的替代方案 |
| --- | --- | --- |
| 全局变量 | 创建整数并在触发器中加一 | 多个对象能读取同一个值 | 每段航道使用独立开关对象 |
| 周期计时器 | 每 0.2 秒触发一次事件 | 可启用、暂停和重置 | 用循环机关或动画事件代替 |
| 范围检测 | 玩家进入圆形区域后点亮灯 | 能检测进入、停留、离开 | 使用多个小触发区拼接 |
| 对象启停 | 进入区域后开启炮台 | 未启用对象不执行攻击 | 预先隔离对象，按关卡传送 |
| 跟随或附着 | 装饰船体跟随玩家单位 | 移动和转向时不明显脱离 | 直接把可控单位外观包装成船 |
| 地形碰撞 | 玩家无法穿过测试岛屿 | 不会卡进模型内部 | 隐形阻挡体或禁行伤害区 |
| 投射物与伤害 | 炮台向玩家发射一枚弹体 | 命中能扣血并销毁弹体 | 射线判定或延时范围伤害 |
| 三选一交互 | 三个选择台只能触发一个 | 选择后另外两个立即关闭 | 三扇门、NPC 对话或踩踏区域 |
| 重置流程 | 玩家死亡后恢复初始状态 | 变量、对象和位置全部重置 | 重新加载独立关卡 |

能力验证完成后，在文档副本中给每一项标记：

- `原生支持`：可以直接实现。
- `需要组合`：需要两个以上组件拼接。
- `使用降级`：采用本文替代方案。
- `暂时删除`：首版不实现，避免拖慢闭环。

## 二、UGC 总体架构

### 2.1 不做真实船舶物理

千星奇域首版不建议实现浮力、真实水面碰撞、复杂载具刚体或多人载具同步。推荐使用“载具假象”：

1. 一个可控制单位作为实际玩家主体。
2. 船体模型作为该单位的外观、附着物或跟随装饰。
3. 玩家单位负责位置、朝向、生命和触发器判定。
4. 船首炮、侧舷炮和尾雷使用独立机关，根据玩家位置和朝向触发。
5. 海面只是视觉场景；岛屿和浅滩使用简单阻挡体与区域效果。

如果编辑器不允许修改玩家移动方式：

- 使用系统自带角色移动。
- 将奔跑或冲刺解释成船舶加速。
- 用船体装饰遮住角色主体。
- 用隐形走道、宽阔水面平台和岛屿阻挡制造航行感。

### 2.2 航段激活架构

长海道不要让所有炮台和检测器一直运行。把地图拆成五个航段：

| 航段 | 建议时长 | 内容 | 激活条件 |
| --- | ---: | --- | --- |
| A 教学段 | 15-25 秒 | 转向、加速、单炮台 | 开局 |
| B 第一次分路 | 30-45 秒 | 红蓝路线、散射炮 | 进入 B 入口 |
| C 中段遭遇 | 35-50 秒 | 狙击炮、弯道、补给 | 离开 B 汇合点 |
| D 第二次分路 | 35-50 秒 | 红蓝金路线、组合火力 | 进入 D 入口 |
| E 精英堡垒 | 40-60 秒 | 精英炮台与出口封锁 | 进入终点海域 |

每个航段建立三个区域：

- `Segment_X_Enter`：启用本段炮台、机关、提示和检测循环。
- `Segment_X_Exit`：关闭上一段的攻击与范围检测。
- `Segment_X_Reset`：失败或重开时恢复本段对象。

原则：同一时间只保持当前航段和前后缓冲航段活跃。

### 2.3 推荐对象清单

建议在编辑器对象层级中采用统一前缀，后续排错会轻松很多：

| 对象前缀 | 示例 | 用途 |
| --- | --- | --- |
| `SYS_` | `SYS_GameFlow` | 全局流程、失败、过关、重置 |
| `SHIP_` | `SHIP_Controller` | 船的位置、朝向、生命、速度 |
| `WPN_` | `WPN_Bow_01` | 船首炮、侧舷炮、尾雷 |
| `SEG_` | `SEG_B_Enter` | 航段激活与关闭 |
| `ISL_` | `ISL_B_03` | 岛屿模型和阻挡 |
| `TUR_` | `TUR_C_Sniper_02` | 岛上炮台 |
| `ROUTE_` | `ROUTE_D_Gold` | 分路入口和奖励 |
| `UPG_` | `UPG_Damage` | 三选一强化台 |
| `UI_` | `UI_LockWarning` | 提示、倒计时和结算 |
| `FX_` | `FX_BowShot` | 发光、命中和锁定特效 |

## 三、变量设计

变量尽量集中管理，不要让每个炮台各自复制一套全局变量。

### 3.1 全局流程变量

| 变量 | 类型 | 初始值 | 用途 |
| --- | --- | ---: | --- |
| `game_state` | 整数/枚举 | `0` | `0准备 1航行 2强化 3失败 4胜利` |
| `route_index` | 整数 | `1` | 当前海道编号 |
| `active_segment` | 整数 | `1` | 当前激活航段 |
| `exit_unlocked` | 布尔 | `false` | 精英炮台死亡后开放出口 |
| `route_choice_1` | 整数 | `0` | 第一处分路选择 |
| `route_choice_2` | 整数 | `0` | 第二处分路选择 |
| `upgrade_choice_locked` | 布尔 | `false` | 防止三选一重复触发 |

### 3.2 航船变量

| 变量 | 初始建议 | 用途 |
| --- | ---: | --- |
| `ship_hp` / `ship_hp_max` | `120` | 船体生命 |
| `ship_speed` | `1.0` | 基础自动航速倍率 |
| `ship_turn_speed` | `1.0` | 转向速度倍率 |
| `ship_boost` | `false` | 是否处于加速 |
| `ship_boost_timer` | `0` | 移动端/手柄可改成限时加速 |
| `ship_shield` / `ship_shield_max` | `0` | 可再生护盾 |

### 3.3 武器与强化变量

| 变量 | 初始建议 | 用途 |
| --- | ---: | --- |
| `bow_damage` | `18` | 船首炮伤害 |
| `bow_interval` | `0.48` | 船首炮间隔 |
| `bow_range` | `310` | 船首炮检测距离 |
| `bow_projectiles` | `1` | 主炮弹数 |
| `bow_pierce` | `0` | 穿透层数 |
| `side_damage` | `11` | 侧舷伤害 |
| `side_interval` | `1.05` | 侧舷间隔 |
| `mine_level` | `0` | 尾雷是否解锁及等级 |
| `shield_level` | `0` | 护盾强化等级 |
| `artillery_count` | `0` | 炮击系强化计数 |
| `defense_count` | `0` | 防御系强化计数 |
| `mobility_count` | `0` | 机动系强化计数 |

### 3.4 炮台公共变量

如果编辑器支持对象局部变量，每座炮台保存：

- `turret_enabled`
- `turret_type`
- `turret_hp`
- `turret_range`
- `turret_lock_progress`
- `turret_fire_cooldown`
- `turret_segment`

如果不支持局部变量，首版不要做动态生成，直接预摆炮台并给每座炮台独立命名。

## 四、具体触发器实现

以下使用“事件 → 条件 → 动作”的通用写法。实际节点名称以编辑器为准。

### 4.1 开局初始化

```text
事件：游戏开始
条件：无
动作：
  设置 game_state = 1
  设置 route_index = 1
  设置 active_segment = 1
  设置 exit_unlocked = false
  设置 ship_hp = ship_hp_max
  清空全部路线选择与强化计数
  传送玩家到海道入口
  重置玩家朝向为北
  启用 SEG_A
  禁用 SEG_B 至 SEG_E
```

### 4.2 船自动航行与转向

优先方案：

```text
事件：每帧或高频周期
条件：game_state == 1
动作：
  读取玩家方向输入
  如果存在方向输入：
    将船头逐渐转向输入方向
  沿船头方向施加基础移动
  如果加速键按下：
    移动倍率使用 boost_multiplier
```

编辑器不支持自定义移动时：

1. 保留系统角色移动。
2. 禁用或弱化角色跳跃。
3. 普通移动表现为巡航，加速键使用冲刺或技能。
4. 船首炮朝向跟随角色面向。
5. 用宽海道与岛群限制移动，不强求持续自动位移。

手柄和移动端建议把“按住加速”改成“点击技能后加速 2 秒，冷却 5 秒”，减少持续按键负担。

### 4.3 船首自动炮

```text
事件：每 0.2 秒更新一次目标
条件：game_state == 1
动作：
  检测船首扇形范围内已启用炮台
  选择距离最近目标
  保存为 bow_target

事件：bow_interval 计时结束
条件：
  bow_target 有效
  bow_target 距离 <= bow_range
动作：
  从船首炮口生成投射物
  投射物朝 bow_target 当前方向飞行
  命中后造成 bow_damage
  重置开火计时器
```

不支持扇形检测时，用三个重叠的前方圆形区域近似。不要每帧遍历地图全部炮台。

### 4.4 左右侧舷炮

1. 在船体左右各放一个短距离范围检测器。
2. 左检测器只控制左炮，右检测器只控制右炮。
3. 检测到目标后，每次发射 2 枚低伤害弹。
4. 两侧使用同一公共冷却，避免同时触发过多弹体。
5. 如果编辑器难以附着检测器，就用“玩家与炮台相对方向”条件判断左右侧。

### 4.5 自动尾雷

首版推荐降级成范围爆炸，不需要真实物理地雷：

```text
事件：尾雷冷却结束
条件：
  mine_level > 0
  船尾检测区存在炮台或追击敌人
动作：
  在船后方生成尾雷视觉物
  等待 0.8 秒
  对尾雷范围内敌人造成范围伤害
  销毁尾雷视觉物
```

限制场上最多存在 3 枚尾雷。

### 4.6 炮台锁定攻击

普通炮台：

```text
事件：玩家进入炮台范围
动作：
  显示锁定线
  lock_progress = 0

事件：每 0.1 秒
条件：玩家仍在范围内
动作：
  lock_progress += 0.1
  更新锁定环表现

事件：lock_progress >= lock_time
动作：
  记录玩家当前位置
  发射直线投射物
  lock_progress = 0
```

三类炮台只改变参数和发射动作：

| 炮台 | 锁定 | 弹体 | 玩家对策 |
| --- | ---: | --- | --- |
| 直射炮 | 中 | 单枚中速弹 | 转向躲避 |
| 散射炮 | 快 | 五枚扇形慢弹 | 提前拉远 |
| 狙击炮 | 慢 | 单枚高速高伤弹 | 看锁定线急转 |
| 精英炮 | 中 | 三枚齐射或交替攻击 | 绕岛、清炮台 |

### 4.7 航段启停

```text
事件：玩家进入 SEG_C_Enter
条件：active_segment < 3
动作：
  active_segment = 3
  启用 C 段炮台和检测区域
  显示 C 段提示
  禁用 A 段炮台
```

“禁用”至少应包括停止锁定计时器和停止生成投射物。纯粹隐藏模型但保留逻辑没有性能收益。

### 4.8 红蓝金路线

每处分路使用一个组编号，例如 `route_group = 1`。

```text
事件：玩家进入 ROUTE_1_RED
条件：route_choice_1 == 0
动作：
  route_choice_1 = 1
  bow_damage *= 1.12
  side_damage *= 1.12
  关闭 ROUTE_1_BLUE 和 ROUTE_1_GOLD
```

| 路线 | 地图特征 | 即时奖励 | 风险 |
| --- | --- | --- | --- |
| 赤潮捷径 | 短、炮台密集 | 本局输出提升 | 交叉火力 |
| 澄蓝稳流 | 长、空间宽 | 修复与机动提升 | 航行时间更长 |
| 辉金秘航 | 隐蔽或限时开放 | 稀有强化/护盾 | 精英炮或窄道 |

奖励必须在进入分路前可见，不能让玩家选完才知道内容。

### 4.9 精英炮台与出口

```text
事件：进入 E 段
动作：
  exit_unlocked = false
  启用精英炮台
  出口显示红色封锁效果

事件：精英炮台死亡
动作：
  exit_unlocked = true
  关闭剩余锁定线
  出口切换为蓝金开放效果

事件：玩家进入出口
条件：exit_unlocked == true
动作：
  game_state = 2
  停止船移动与全部攻击计时器
  显示三选一强化
```

### 4.10 三选一强化

如果支持自定义 UI，可直接展示三个按钮；否则使用三个实体选择台。

推荐实体方案：

1. 把玩家传送到安全港或让船停在强化区域。
2. 随机启用三个强化台。
3. 每个强化台显示名称、系别和效果。
4. 玩家进入其中一个选择区。
5. 先设置 `upgrade_choice_locked = true`。
6. 应用强化并关闭三个选择台。
7. 增加对应流派计数。
8. 检查是否达到 3 层共鸣。
9. `route_index += 1`，重置航段并开始下一段海道。

随机系统不稳定时，首版可以使用固定轮换池：

- 第 1 次：伤害、护盾、机动。
- 第 2 次：射速、侧舷、尾雷。
- 第 3 次：穿透、恢复、干扰。

### 4.11 失败与重开

```text
事件：ship_hp <= 0
动作：
  game_state = 3
  停止移动、炮台计时器和弹体生成
  清理现有投射物
  显示失败与重开交互

事件：选择重开
动作：
  恢复全部初始变量
  重置所有航段对象
  清除强化和路线选择
  传送玩家到入口
  game_state = 1
```

## 五、推荐制作步骤

### 阶段 0：能力验证

- 完成九项能力检查。
- 决定船体采用附着物、替换外观还是普通角色伪装。
- 决定攻击采用投射物、射线还是范围伤害。
- 产出结果：一张能力支持表。

### 阶段 1：最小航行闭环

- 只搭一个屏幕海面和两座岛。
- 完成船移动、转向、加速、岛屿阻挡。
- 完成一个直射炮台及锁定预警。
- 完成船首自动攻击。
- 产出结果：玩家能航行、躲避、攻击、死亡和重开。

### 阶段 2：单段关卡

- 做一段 30 至 45 秒海道。
- 加入散射炮与狙击炮。
- 加入一个红蓝分路。
- 在终点放精英炮台和出口。
- 产出结果：一段完整可重复游玩的海道。

### 阶段 3：强化闭环

- 加入固定三选一，不先做随机。
- 完成伤害、护盾、机动三个基础强化。
- 验证强化进入下一段后仍有效。
- 加入三个同系强化触发共鸣。
- 产出结果：至少可以连续完成三段航行。

### 阶段 4：长地图与性能

- 把地图扩展到五个航段。
- 加入航段启停，不允许全图逻辑同时运行。
- 限制弹体、锁定线和尾雷数量。
- 测试最低目标设备和多人房间条件。
- 产出结果：6 至 8 分钟完整单局。

### 阶段 5：视觉与发布

- 替换测试方块和临时特效。
- 统一红色危险、蓝色安全、金色稀有的视觉语言。
- 添加开局 15 秒无文字或少文字教学。
- 检查失败、重开、断线、卡地形和重复触发。
- 完成标题、封面、玩法说明和版本号。

## 六、UGC 性能与稳定性

首版建议预算：

| 项目 | 建议上限 |
| --- | ---: |
| 同时激活炮台 | 8-12 |
| 同时存在敌方弹体 | 30-40 |
| 同时存在玩家弹体 | 20-30 |
| 同时存在尾雷 | 3 |
| 高频周期检测器 | 10 个以内 |
| 目标刷新频率 | 0.15-0.25 秒 |
| 单局时长 | 6-8 分钟 |

优化原则：

- 目标选择不需要每帧刷新。
- 炮台离开激活航段后立即停止逻辑。
- 锁定线只对当前锁定玩家显示。
- 弹体超时或离开战区后立即销毁。
- 不使用大量透明叠层和全图粒子。
- 不依赖精确物理碰撞完成关键流程。
- 关键变量变更后再更新 UI，不要每帧重建 UI。

多人扩展前，先确认伤害、路线选择、强化归属和船体移动由哪一端判定。首版推荐单人，避免同步问题把制作复杂度翻倍。

### 6.1 核心流程摘要

1. 开局初始化变量：`route = 1`、船体生命、船武器数值、护盾层数、炮台强度倍率。
2. 开始当前海道：生成长海图、边界岛群、分流岛群和岛上炮台，把船放在下方入口，把出口放在最上方。
3. 船移动循环：船持续沿当前船头方向自动航行，玩家用上下左右输入改变船头朝向；按住加速键时提高航速，松开后恢复基础速度。松开方向输入时保持现有航向。
4. 船自动攻击循环：船首主炮检测前方目标，侧舷炮检测左右目标，尾雷检测船后目标；船身随左右操舵轻微转向，朝向决定输出方式。
5. 炮台锁定循环：船进入炮台射程后显示瞄准线，锁定完成后发射直射弹、散射弹或高速狙击弹。
6. 地形限制循环：岛屿作为硬阻挡，玩家撞上会被推出岛屿边缘，需要及时转舵离开。
7. 炮台死亡时：移除炮台并降低当前海道压力；过关不要求清掉全部炮台。
8. 玩家经过分路信标：红色捷径增加输出，蓝色稳流修复船体，金色秘航给予稀有奖励；同一分路只能选一个。
9. 玩家抵达终点前必须击破精英炮台，解除出口封锁。
10. 如果玩家进入已开放的出口区域：停止航船和战斗循环，显示三个强化选择。
11. 玩家选择强化：只应用一个强化，隐藏全部选择，`route += 1`，进入下一段海道。
12. 玩家船体生命归零：停止航船和战斗循环，显示失败/重开。

### 6.2 视觉方向

整体可以贴近原神式明亮幻想、清透材质、元素感、柔和发光和轻量粒子，但必须保持原创，不直接复刻具体角色、怪物、武器、命座图标或 UI。

| 资产 | 接近原神的质感方向 | 原创差异点 |
| --- | --- | --- |
| 航船 | 清晰轮廓、金色描边、发光核心 | 星帆航船，核心像航海罗盘和星轨炮塔 |
| 炮塔 | 元素发光、攻击特效清楚 | 星轨主炮、辉光连弹、彗星穿透 |
| 炮台 | 元素机关、清晰锁定反馈 | 岛上星轨炮台，不使用已知怪物或官方机关轮廓 |
| 弹幕 | 颜色区分清楚、命中反馈明显 | 星尘炮弹、粉色星渊飞弹 |
| 航线信标 | 高亮元素区域、远距离可读 | 赤潮、澄蓝、辉金三种原创航路标记 |
| 强化选择 | 类似神龛/机关的可读交互 | 三个星标信标、甲板选择台或港口门扉 |
| 场地 | 明亮幻想天空/海面 | 岛屿海道、星潮海面、奇域远航氛围 |
| 地形 | 可读的幻想地貌和元素危险提示 | 岛屿、浅滩、航道边界，不做真实岩石写实风 |

需要避免：直接仿制官方角色脸、服装剪影、武器轮廓、命座图案、七元素图标、官方 UI 图标。

### 6.6 当前推荐地图结构

首版做一张约 5 个屏幕长度的纵向海图：

1. 船从下方入口自动出发，玩家用 `WASD` 控制航向，用 `Shift` 主动加速。
2. 第一段用中央大岛把海道分成左右两路：一边更短但有炮台，另一边更长但压力较低。
3. 中段由错位岛群形成连续弯道，并提供两到三个可通过的缝隙，避免只有唯一解。
4. 第二处分流围绕堡垒岛展开，外侧安全路线和中央高风险路线最后重新汇合。
5. 岛上炮台显示射程，船进入射程后开始锁定并攻击。
6. 船自动攻击射程内炮台，玩家通过转向、回撤和加速时机控制风险；加速能更快脱离锁定区，但也会扩大转弯半径并压缩路线判断时间。
7. 玩家到达最上方出口后过关，清不清炮台由玩家自己取舍。
8. 每段海道结束后暂停并出现三个强化信标。

地图设计原则：

- 长度约为 4 至 6 个屏幕，保证强化后的航行体验有发挥空间。
- 每个分流至少有两条可通路线，并在 1 至 2 个屏幕后重新汇合。
- 短路线增加炮台覆盖或交叉火力，长路线减少火力但消耗更多航行时间。
- 不做纯直线走廊；岛群中心线左右交替，让路线自然形成 S 弯。
- 后续可加入浅滩减速、可破坏补给炮台、精英炮台和隐藏支路。

## 七、后续扩展方案

扩展顺序应优先增加“新决策”，不要只增加地图长度和炮台数量。

### 7.1 第一阶段：低成本内容扩展

这些内容主要复用现有变量和触发器，最适合首版上线后增加：

**新航道环境**

- 星潮区：进入后逐渐恢复护盾。
- 辉风区：提升航速和转向，但敌方狙击炮更密集。
- 雷云区：周期标记落雷区域，命中玩家和炮台。
- 浅滩区：降低速度，迫使玩家提前选择路线。

实现方式：每种环境都是一个区域触发器，进入时修改倍率，离开时恢复。区域效果必须有明显地面颜色和边界。

**新炮台组合**

- 护盾塔：附近炮台获得减伤，优先摧毁。
- 灯塔：扩大其他炮台锁定范围，本身不攻击。
- 修复塔：周期恢复精英炮台生命。
- 地雷塔：在海道固定位置生成危险区。

实现方式：尽量使用“范围增益 + 周期事件”，不需要复杂 AI。

**小型事件**

- 20 秒闸门防守。
- 限时穿过金色航标。
- 摧毁补给塔恢复船体。
- 二选一：救援补给船或走捷径。

实现方式：进入事件区域后设置 `event_active`，完成条件后开放出口或给予奖励。

### 7.2 第二阶段：强化质变

在基础数值强化稳定后，加入容易用条件分支表达的质变：

| 质变强化 | 效果 | UGC 实现 |
| --- | --- | --- |
| 星爆主炮 | 主炮命中后产生小范围爆炸 | 命中事件追加一次范围伤害 |
| 雷弧侧舷 | 左右侧舷都命中时追加连锁伤害 | 记录左右命中时间，短时间内同时成立则触发 |
| 潮汐雷场 | 连续布置三枚尾雷后一起引爆 | `mine_count == 3` 时执行范围伤害 |
| 破盾齐射 | 护盾归零时立即触发主炮齐射 | 监听护盾从 1 变为 0 |
| 星轨冲锋 | 加速期间提高射速 | `ship_boost == true` 时修改开火间隔 |
| 航线大师 | 选择危险路线后下一次强化升级 | 路线变量影响强化应用倍率 |

质变强化建议每局最多出现 2 至 3 个，避免触发器组合爆炸。

### 7.3 第三阶段：精英与首领

首领不需要自由移动 AI，可以继续使用岛上机关：

1. 第一阶段：左右炮台交替锁定。
2. 第二阶段：中央核心开启护盾，需要摧毁两座灯塔。
3. 第三阶段：航道出现周期横扫攻击。
4. 玩家击破核心后开放最终出口。

这样能保留首领战节奏，又不依赖复杂寻路。

首领状态变量：

- `boss_phase`
- `boss_hp`
- `boss_shield`
- `left_tower_alive`
- `right_tower_alive`
- `boss_attack_index`

### 7.4 第四阶段：地图随机化

不要先做程序化随机岛屿。UGC 更适合“模块随机”：

1. 预先制作 6 至 10 个航段模块。
2. 每局从同难度模块中选择一个启用。
3. 模块入口和出口保持统一位置。
4. 每个模块拥有自己的炮台组、路线奖励和重置触发器。

这种方式更容易保证可通行性，也方便逐个测试。

### 7.5 第五阶段：多人合作

多人不是首版目标。准备扩展时可选两种方案：

**同船合作**

- 一名玩家控制航向。
- 其他玩家负责选择强化、触发副炮或修复。
- 优点是地图规则基本不变。
- 难点是载具控制权、断线和角色跟随。

**多船合作**

- 每名玩家控制一艘小船。
- 炮台按最近玩家或固定目标攻击。
- 路线选择采用队伍投票或队长决定。
- 难点是弹体、伤害和目标检测数量成倍增加。

多人实现前必须先验证：

- 玩家移动和伤害由主机还是服务器判定。
- 强化是个人变量还是队伍变量。
- 玩家死亡后复活还是全队失败。
- 分路时玩家分散是否会同时激活两个航段。

推荐先做同船双人辅助玩法，再考虑多船。

## 八、不建议首版实现

- 真实海浪、浮力和船体物理。
- 全图动态生成岛屿。
- 大量可自由移动的海上敌舰。
- 同时存在几十种状态效果。
- 四人多船并带独立强化。
- 依赖精准弹体反弹或复杂碰撞的核心机制。
- 大量随机强化互相组合的无限模式。

这些内容并非不能做，而是会显著增加 UGC 触发器数量、同步风险和测试成本。首版应先证明“自动航行、路线决策、自动战斗、三选一构筑”本身好玩。

## 九、发布前测试清单

### 核心闭环

- 船在没有输入时仍沿船头航行。
- 四方向输入能够稳定改变航向。
- 加速能开启、结束并正确恢复基础速度。
- 船不会穿过岛屿，也不会永久卡在岛边。
- 三种炮台攻击前都有可辨认预警。
- 玩家武器能正确区分船首、侧舷和船尾。
- 分路只能选择一条，奖励不会重复领取。
- 精英死亡后出口一定开放。
- 三选一只能选择一个，强化下一段仍然有效。
- 失败重开后所有变量和对象恢复初始状态。

### UGC 稳定性

- 离开旧航段后旧炮台不再攻击。
- 弹体不会永久留在场景中。
- 快速反复进入触发区不会重复结算。
- 玩家在强化界面和失败界面不能继续移动。
- 最低目标设备上没有明显掉帧。
- 手柄、键鼠和移动端至少有一种完整可用操作方案。

### 视觉与版权

- 危险、奖励、出口和锁定提示能在战斗中快速辨认。
- 不直接复刻原神具体角色、武器、命座、元素图标或官方 UI。
- 使用千星奇域允许的官方素材时遵守其发布与创作规则。
- 自制资产保持星海航船、星轨炮台和奇域远航的原创识别度。

## 十、推荐首版范围

首个可发布版本建议只包含：

- 1 张由 5 个航段组成的长海道。
- 2 次分路选择。
- 3 类普通炮台和 1 个精英炮台。
- 船首炮、侧舷炮和尾雷。
- 9 至 12 个强化。
- 3 类构筑共鸣。
- 约 6 至 8 分钟单局。
- 单人模式。

达到这个范围后先让玩家试玩，重点观察：

- 玩家是否理解船会自动前进。
- 玩家是否愿意主动使用加速。
- 分路奖励是否真的影响选择。
- 船首、侧舷和尾雷是否产生不同走位。
- 三选一是否能形成明显构筑。
- 玩家在哪个航段最容易觉得重复或拖沓。

根据试玩数据再决定优先增加新航段、新强化还是首领战。
